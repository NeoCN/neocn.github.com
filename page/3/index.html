Liferay Portal 换用 LGPL 许可证

Mar 11th, 2010 | Comments

从 Liferay Portal  6.0开始，将在Lesser GNU Public License (LGPL) v2.1 下进行发布。

Liferay Portal 之前一直是采用比较宽松的MIT License。

官方博客提到，MIT许可证带来了很多好处，但是也有很多问题，比如一些公司采用并修改Liferay Portal代码后打包成产品发布，但是却不回馈Liferay Portal社区，这一方面吞噬着Liferay Portal的市场，另一方面也伤害了Liferay Portal社区的感情。这些将Liferay Portal打包进行销售的公司，本来是可以成为Liferay的客户的，但是，现在的情况是这些公司销售的产品中90%的代码来自于Liferay，但却不需支付任何费用，而且对Liferay Portal所做的修改，也可以不提交给Liferay Portal。

这再一次体现了基于信任的开源模型的缺点。

开源社区提问的10条黄金法则

Mar 10th, 2010 | Comments

  1. 要有礼貌
  2. 尽可能的描述准确、简短
  3. 提问前做一做功课 (Google, 搜索新闻组,阅读教程,…)
  4. 尽量每次只问一个问题
  5. 请在写完后校对一遍你的问题
  6. 不要指望别人做你的工作
  7. 避免私人电子邮件，如果有其他渠道存在的话
  8. 提醒自己别人是自愿无偿提供帮助的
  9. 阅读并实践他人提供的建议
  10. 如果你有解决方案请与大家共享
以下是英文原文：
  1. Don’t be rude
  2. Try to be as precise and as short as possible
  3. Do your homework before asking (Google, search newgroups, read tutorials,…)
  4. Try to ask one question at the time
  5. Proof-read your question after writing it
  6. Don’t expect others to do your work
  7. Avoid private email if other channels exists
  8. Remind yourself that people help you voluntary
  9. Read and follow up on suggestions
  10. If you get a solution let others know

Feb 17th, 2010 | Comments

前段时间尝试搭建虚拟机环境，综合利用一下服务器资源，在物理机上安装了VMware Workstation 7，虚拟机是Windows XP SP2 ，共5个虚拟机，机器名为PC1～PC5，虚拟机网络使用的是桥接模式；物理机上安装了Oracle 10g数据库，虚拟机PC1上安装WebLogic 10.3 ；在使用中出现了下面的现象：

  1. 在虚拟机PC1上启动WebLogic 10.3，新建JDBC DataSource，在测试连接时，无法连接到宿主物理机上的Oracle 10g数据库；

Oracle EXP-00003 异常解决办法

Jan 12th, 2010 | Comments

今天下午，想在测试环境下进行数据库迁移的模拟测试，本机Oracle 版本为9.2.0.1 ，服务器Oracle版本为9.2.0.8 ，在使用EXP命令进行用户数据导出时，如果表中有数据类型是Lob的列，则会出现EXP-00003：未找到段×××的存储定义(EXP-00003 : no storage definition found for segment …..)。

出现 EXP-00003异常，是Oracle数据库的一个Bug。当使用低版本的EXP从高版本(9.2.0.5或者更高)去导出含有Lob列的表的时候，就会遇到EXP-00003异常。

为了解决这个问题，可以选择升级客户端，将客户端EXP升级到较高版本，可以解决这个Bug；当然有些人不想这么麻烦，那么可以用下面的临时解决方案(替换视图”exu9tne”)：

EXP导出前，在SYS用户下执行下面的SQL：

CREATE OR REPLACE VIEW exu9tne (
tsno, fileno, blockno, length) AS
SELECT ts#, segfile#, segblock#, length
FROM sys.uet$
WHERE ext# = 1
UNION ALL
SELECT * FROM SYS.EXU9TNEB
/

EXP导出完成后，执行下面的SQL恢复视图定义(Metalink建议)：

CREATE OR REPLACE VIEW exu9tne (
tsno, fileno, blockno, length) AS
SELECT ts#, segfile#, segblock#, length
FROM sys.uet$
WHERE ext# = 1
/

破解Excel的工作簿和工作表保护

Dec 26th, 2009 | Comments

Excel的工作簿和工作表的保护口令，保存的是经过转换的密码，用户输入的任何长度的保护口令都会被转换为12字符长的字符串，其中前11个字符均是某两个特定字符中的一个，最后一位有95种可能值，那么，这个转换后的保护密码共计有 2^11×95 = 194560 种组合。

这样的组合对现在的计算机来说只是小儿科，几秒钟就可以全部尝试一遍，给出两个比较数字：一个包含小写字母的4字符长度的密码，总计有456976种组合；一个包含小写字母、大写字母和数字的3字符长度的密码，将会有238328种组合。

强调一下：无论初始密码是什么，194560个组合中的一个，将解锁受保护的工作簿或者工作表。

Java EE 6 发布！

Dec 10th, 2009 | Comments

在过去的几个月里一些Java EE 6相关的JSRs((Java Specification Requests))纷纷定稿，最后一轮投票与11月30日结束并且所有均获通过。今天，2009年12月10日，Java EE 6的参考实现 —— Java EE 6 和 GlassFish v3，发布了。

在Java EE 5发布4年半之后，我们进入了下一个Java EE 时代。GlassFish v3是第一个完全支持所有Java EE 6技术的应用服务器。支持的技术包括但不局限于下列技术：

  • Servlet 3.0
  • JSF 2.0
  • WebBeans
  • CDI (Contexts and Dependency Injection)
  • Bean Validation
  • EJB 3.1
  • JPA 2.0
  • JAX-RS
GlassFish v3可以以两种方式下载：第一种包含所有Java EE 6 技术，第二种包括所有Java EE 6 Web Profile标准中的技术。

值得一提的是，Java EE比以前更强大了。现在我们可以使用注解来代替XML配置，我们可以不使用EAR文件因为WAR已经足够了，Ajax已经被标准化，从客户端甚至到数据库都支持异步调用。

iText License 更换为 AGPL

Dec 8th, 2009 | Comments

著名的Java PDF类库iText于12月7日发布了5.0版，新版本除了更改了包名、要求JDK5.0、修复Bug外，最主要的一个改变是其License的改变，许可由MPL/LGPL更换为AGPL。 

Changes made by Bruno : Changed the license from MPL/LGPL to AGPL
对于GPL、LGPL是比较常见的，这个AGPL到底有什么约束呢？我们一般情况下会把iText作为类库来使用。

GPL许可要求任何使用GPL代码的软件也必须在GPL许可下发布，提供源代码；LGPL则允许在只对LGPL许可的代码做静态引用，即作为类库调用的情况下，不开源代码；而AGPL，其实是对GPL的一个补充，因为GPL许可下的代码和软件，在用作提供服务的情况下，比如用于提供Web Service服务，是不受GPL许可约束的，AGPL即是针对这种情况增加了约束条款。

以下内容摘录自网络： 

AGPL这个协议的制定是为了避免一个GPL/LGPL协议中的漏洞，称之为 Web Service Loopwhole。
这主要是由于 GPL是针对传统的软件分发模式的商业模式(以微软为代表), 如果你使用的GPL的代码作为
基础完成你自己的软件，如果你要分发你的软件，你的软件必须也是GPL的。随着以Google为代表的软件
作为服务的互联网公司的兴起，它们的“不分发软件，为客户提供网络服务”的商业模式就不受GPL协议的
约束，所以Google公司在构筑他的搜索引擎的时候可以随心所欲的拿现有的GPL协议的开源代码，无需开
源他的修改成果。AGPL协议在GPL协议的基础上加上了这个约束。
所以，iText新的开源许可，对于商业应用来讲，比GPL许可更不友好。
